# 🛠️ 软件工程速查卡

## 第一层：核心概念速查表

| 概念 | 定义 | 记忆技巧 | 易混点 |
|------|------|---------|--------|
| 瀑布模型 | 线性阶段化，前一阶段完成后才进入下一阶段 | 需求→设计→实现→测试→部署 | 缺乏灵活性，难以应对变更 |
| 迭代模型 | 分多个迭代周期，每个周期完成完整的开发过程 | 每个迭代产出可运行的软件 | 比瀑布灵活，成本可控 |
| 敏捷开发 | 强调快速迭代、客户参与、团队协作 | Scrum、XP、Kanban 等框架 | 文档少，代码质量依赖技能 |
| 原型法 | 快速构建可用系统原型，与用户反复确认 | 用于需求不清或快速验证 | 不是最终产品，易被当成产品 |
| 需求分析 | 收集、整理、验证用户需求，产出需求规格说明书 | SRS (Software Requirement Specification) | 需求变更管理很重要 |
| 架构设计 | 确定系统的总体结构、模块、接口、数据流 | SAD (System Architecture Document) | 架构决定性能和可维护性 |
| 单元测试 | 测试单个函数/方法的功能正确性 | 由开发人员编写，白盒测试 | 覆盖率 > 80% 目标 |
| 集成测试 | 测试多个模块协同工作的功能 | 黑盒测试，测试模块间接口 | 需定义集成顺序 |
| 系统测试 | 针对整个系统的功能、性能、安全等综合测试 | 黑盒测试，接近真实环境 | 包括回归测试 |
| 代码评审 | 开发人员互相审查代码质量、安全、规范性 | 静态分析，发现潜在缺陷 | 提高代码质量的有效手段 |
| 配置管理 | 管理软件版本、变更、发布、依赖关系 | 版本控制、分支策略、发布计划 | Git、SVN 等工具支持 |
| 文档管理 | 编写和维护需求、设计、测试等各阶段文档 | 需求文档、设计文档、测试报告 | 文档应同步更新 |
| 软件质量 | 符合需求的能力，包括功能性、可靠性等 | ISO/IEC 25010 标准 | 质量管理贯穿整个过程 |

## 第二层：常考公式 / 对比表

### 开发模型对比

| 模型 | 特点 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|--------|
| **瀑布** | 线性、阶段化 | 计划清晰、管理简单 | 缺乏灵活性、响应慢 | 需求稳定的大型项目 |
| **迭代** | 循环迭代、增量交付 | 灵活、易调整、提早交付 | 计划复杂、版本众多 | 需求可能变化的项目 |
| **敏捷** | 短迭代、自组织、持续反馈 | 高效、响应快、质量好 | 文档少、风险高 | 创业、初创、创新产品 |
| **原型** | 快速构建、用户验证 | 需求澄清快、降低风险 | 可能成为最终产品 | 需求不清或技术探索 |
| **V 模型** | 测试与开发并行 | 质量保证充分 | 过程复杂、成本高 | 质量要求极高的项目 |

### 测试分类与对比

| 测试类型 | 对象 | 方法 | 时机 | 目标 |
|--------|------|------|------|------|
| **单元测试** | 函数/方法 | 白盒 | 编码完成 | 逻辑正确性 |
| **集成测试** | 模块间接口 | 黑盒/灰盒 | 单元测试后 | 接口兼容性 |
| **系统测试** | 整个系统 | 黑盒 | 集成后 | 整体功能、性能、安全 |
| **验收测试** | 用户需求 | 黑盒 | 上线前 | 满足需求程度 |
| **回归测试** | 变更后的系统 | 黑盒/白盒 | 每次修改后 | 防止新 Bug 引入 |

### 质量与可靠性

| 指标 | 定义 | 计算公式 |
|------|------|--------|
| 可用性 | 系统正常工作的占比 | A = MTBF / (MTBF + MTTR) |
| 平均故障间隔 | 两次故障之间平均时间 | MTBF = 总工作时间 / 故障次数 |
| 平均修复时间 | 修复故障平均耗时 | MTTR = 总修复时间 / 故障次数 |
| 缺陷密度 | 单位代码中缺陷个数 | 缺陷数 / KLOC（每千行代码） |

## 第三层：高频题型示例

### 题型 0️⃣：开发模型选择

**题目示例**：新产品开发，需求不清楚，需要频繁调整，应该选择什么开发模型？

**答案框架**：
```
分析需求特点：
✓ 需求不清楚 → 需要与用户反复沟通
✓ 频繁变更 → 需要灵活的过程
✓ 快速上市 → 需要提早交付

模型选择：
1. 原型法：快速澄清需求（推荐！）
2. 敏捷开发：快速迭代、持续反馈（推荐！）
3. 迭代模型：增量式交付
❌ 瀑布模型：不灵活，不适合

答案：原型法或敏捷开发
```

---

### 题型 1️⃣：测试计划与策略

**题目示例**：如何设计一个完整的测试计划？包含哪些内容？

**答案框架**：
```
测试计划应包含 5 部分：

1. 测试目标
   - 功能正确性
   - 性能指标
   - 可靠性、安全性

2. 测试范围
   - 包括什么模块
   - 不测试什么（第三方库）

3. 测试策略
   单元 → 集成 → 系统 → 验收
   目标覆盖率 > 80%

4. 测试资源
   - 人员、环境、工具

5. 测试进度
   - 时间表、里程碑
```

---

### 题型 2️⃣：需求追踪矩阵

**题目示例**：什么是需求追踪矩阵？为什么需要？

**答案框架**：
```
需求追踪矩阵（RTM）：
- 追踪需求 → 设计 → 代码 → 测试的对应关系
- 确保需求完整实现和充分测试
- 检测遗漏或不相关的代码

建立方式：
| 需求ID | 需求描述 | 设计模块 | 代码位置 | 测试用例 |
|--------|--------|---------|--------|---------|
| REQ001 | 用户登录 | AuthMod | auth.py | TC001 |
```

### 题型 3️⃣：代码评审

**题目示例**：代码评审的目的和关键内容是什么？

**答案框架**：
```
代码评审目的：
✓ 发现缺陷（逻辑、安全、性能）

评审内容：

```

### 题型 5️⃣：风险管理

**题目示例**：如何识别和应对项目风险？

**答案框架**：
```
风险管理流程：

1. 识别风险
   - 技术风险、资源风险、时间风险
   - 需求风险、外部风险

2. 评估风险
   风险等级 = 概率 × 影响

3. 应对策略
   - 避免（Avoid）
   - 缓解（Mitigate）
   - 接受（Accept）
   - 转移（Transfer）

4. 监控
   - 定期评审风险状态
```

### 题型 6️⃣：成本与进度

**题目示例**：如何估算项目成本和工期？

**答案框架**：
```
估算方法：

1. 自下而上（Bottom-up）
   分解 → 逐项估算 → 汇总
   优：准确；缺：耗时

2. 自上而下（Top-down）
   先估总成本 → 分配各部分
   优：快速；缺：准确度低

3. 类比估算
   参考类似项目

4. 参数估算
   公式：工作量 = a × 代码量^b

计算：
工期 = (估算工作量 + 缓冲) / 可用人力
```

### 题型 7️⃣：可靠性指标

**题目示例**：如何评价系统可靠性？常用指标？

**答案框架**：
```
可靠性指标：

1. MTBF（平均故障间隔）
   MTBF = 总工作时间 / 故障次数

2. MTTR（平均修复时间）
   MTTR = 总修复时间 / 故障次数

3. 可用性（Availability）
   A = MTBF / (MTBF + MTTR) × 100%

例：
- MTBF = 100 小时
- MTTR = 5 小时
- A = 100/(100+5) = 95.24%
```

### 题型 8️⃣：文档管理

**题目示例**：项目需要哪些关键文档？

**答案框架**：
```
关键文档：

1. 需求文档（SRS）
   - 功能需求、非功能需求
   - 用户故事、验收标准

2. 设计文档（SAD）
   - 架构设计、数据库设计
   - 接口设计

3. 编码规范
   - 命名、格式、注释

4. 测试文档
   - 测试计划、测试用例
   - 测试报告、缺陷列表

5. 发布文档
   - 版本说明、变更日志
   - 部署手册
```

### 题型 9️⃣：质量保证

**题目示例**：QA 和 QC 的区别？如何进行质量审计？

**答案框架**：
```
QA vs QC：

QA（质量保证）：
  - 建立流程和标准（预防）
  - 例：制定代码规范

QC（质量控制）：
  - 通过测试发现缺陷（检测）
  - 例：单元测试

质量审计内容：
- 流程遵循
- 文档完整
- 代码质量
- 缺陷管理

审计指标：
- 缺陷密度 < 3/KLOC
- 逃逸率 < 5%
- 覆盖率 > 80%
```
