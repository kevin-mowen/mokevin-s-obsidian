## 第一层：性能指标

| 指标 | 定义 | 单位 | 目标 |
|------|------|------|------|
| **响应时间(RT)** | 用户请求到收到响应的时间 | ms | < 200ms |
| **吞吐量(TPS)** | 单位时间内完成的事务数 | 次/秒 | 越高越好 |
| **可用性(Availability)** | 系统可正常工作的时间占比 | % | > 99.9% |
| **QPS** | 每秒查询数 | 次/秒 | 与系统设计有关 |
| **并发数** | 同时处理的请求数 | 个 | 与硬件、架构有关 |
| **CPU利用率** | CPU工作时间占比 | % | 60-70%最优 |
| **内存利用率** | 已用内存占总内存比例 | % | 70-80%较优 |
| **磁盘I/O** | 磁盘读写速度 | MB/s | 与硬件有关 |

## 第二层：性能优化策略

### 应用层优化

| 优化方法 | 描述 | 效果 |
|--------|------|------|
| **缓存** | 将热数据存在内存(Redis等) | 10-100倍 |
| **异步处理** | 非关键操作异步执行(MQ) | 降低响应时间 |
| **数据库优化** | 索引、分库分表、查询优化 | 5-10倍 |
| **代码优化** | 算法优化、减少循环 | 2-5倍 |
| **连接池** | 复用数据库连接 | 减少建立连接开销 |

### 架构层优化

| 优化方法 | 描述 | 效果 |
|--------|------|------|
| **负载均衡** | 分散请求到多个服务器 | 水平扩展 |
| **读写分离** | 主从复制，读从库 | 提高读性能 |
| **分库分表** | 数据分片，避免单点 | 突破数据库瓶颈 |
| **CDN** | 边缘节点缓存静态资源 | 降低延迟 |
| **消息队列** | 缓冲突发流量 | 削峰填谷 |
| **服务网格** | 微服务通信管理 | 提高可靠性 |

### 基础设施优化

| 优化方法 | 描述 | 效果 |
|--------|------|------|
| **CPU升级** | 更高主频、更多核心 | 提高计算性能 |
| **内存升级** | 更大内存容量 | 缓存更多数据 |
| **SSD** | 替代HDD | 100倍I/O性能提升 |
| **万兆网络** | 更高带宽 | 减少网络延迟 |

## 第三层：系统配置

### 高可用设计（3个9 = 99.9%）

```
99.9% 可用性 = 年停机时间 ≤ 8.76 小时
99.99% 可用性 = 年停机时间 ≤ 52.6 分钟
99.999% 可用性 = 年停机时间 ≤ 5.26 分钟

实现方式：
✓ 冗余：多个副本，单点故障不中断服务
✓ 故障转移：自动检测和切换
✓ 监控告警：及时发现问题
✓ 灾备方案：定期备份和恢复测试
```

### 容量规划

```
计算公式：
所需服务器数 = 峰值流量 / 单台服务器处理能力 + 冗余系数(通常1.5-2)

例：
- 峰值QPS = 10000
- 单台TPS = 1000
- 冗余系数 = 1.5
- 所需服务器 = (10000/1000) × 1.5 = 15 台
```

### 可扩展性考虑

| 方向 | 方案 | 优点 | 限制 |
|------|------|------|------|
| **垂直扩展** | 升级单台硬件 | 简单 | 成本高、有天花板 |
| **水平扩展** | 增加服务器数量 | 成本低、无限制 | 复杂度高、需要考虑分布式 |

## 第四层：常见题型

### 题型1：性能问题诊断

```
现象：系统响应慢

诊断步骤：
1. 检查CPU/内存/磁盘I/O使用率
2. 分析慢查询日志
3. 检查网络延迟
4. 分析应用日志
5. 压力测试定位瓶颈

可能原因：
- CPU高 → 算法问题或并发处理
- 内存高 → 内存泄漏或缓存未清理
- 磁盘高 → 数据库查询慢或磁盘故障
- 网络高 → 带宽不足或分布式调用多
```

### 题型2：容量规划

```
已知：
- 预期日均用户数：100万
- 高峰期时间：10:00-12:00，占日流量的30%
- 平均QPS：10

求：所需服务器数

计算：
- 高峰期QPS = (日均用户 × 30% × 平均QPS) / (高峰期秒数)
  = (100万 × 30% × 10) / 7200 ≈ 417 QPS
- 单台服务器处理能力：100 QPS
- 所需服务器 = 417 / 100 × 1.5 = 6.25 → 7台
```

### 题型3：高可用架构

```
要求：设计一个99.99%可用的系统

方案：
1. 应用层：负载均衡 + 多台应用服务器
2. 数据库：一主多从 + 读写分离
3. 缓存层：Redis集群 + 主从复制
4. 存储：RAID + 磁盘冗余
5. 监控：实时告警，故障自动转移
6. 备份：异地备份，定期恢复测试
```

## 常考公式汇总

```
1. 可用性 = (总时间 - 故障时间) / 总时间
2. MTBF = 总工作时间 / 故障次数
3. MTTR = 总维修时间 / 故障次数
4. 系统可用性 = MTBF / (MTBF + MTTR)

5. 所需容量 = 峰值流量 / 单台处理能力 × 冗余系数
6. 峰值QPS = 日均QPS × 峰值因子
7. 响应时间 = 处理时间 + 网络延迟
8. 吞吐量 = 并发数 / 平均响应时间
```
