# 存储边界规则

> 三件套（PG / Redis / MinIO）+ Qdrant 的分工。**判错存储 = 后期重构**，必须一次搞清。

## 核心规则

| 数据特征 | 用什么 | 例子 |
|---|---|---|
| **结构化、要 SQL 查、关系约束** | [[PostgreSQL]] | 用户表、kb 表、node 表、对话表 |
| **可丢失、要快、过期会自动消失** | [[Redis]] | JWT session、热点缓存、限流计数 |
| **二进制大文件** | [[MinIO]] | .docx / .pdf / .jpg / .zip |
| **向量** | [[Qdrant]]（经 [[Raglite]]） | chunk embedding |

## 判别诀窍

```
┌─────────────────────────────────────────────────────┐
│ Q: 这个数据应该存哪？                                │
├─────────────────────────────────────────────────────┤
│ ① 是个文件吗（图片/文档/压缩包）？      → MinIO     │
│ ② 是个向量吗？                          → Qdrant   │
│ ③ 丢了用户骂街吗？                      → PG       │
│ ④ 丢了用户无感、要毫秒级读吗？          → Redis    │
└─────────────────────────────────────────────────────┘
```

## 一个文件的"指针 vs 数据"模型

```
PG.nodes 表                MinIO 桶
─────────                  ─────────
id: 123                    s3://panda-wiki/docs/123.md
file_url: s3://...   ◄────┘
content: "..."            （文件本体）
```

PG 存"我是谁、属于谁、有什么属性"；MinIO 存"实际字节"。

## 一个文档向量化的"数据三家"

```
PG.nodes (元数据)        Qdrant (向量)         MinIO (原始文件)
────────────────         ─────────────         ──────────────────
id, kb_id, title    →    chunk vec + payload   .md / .docx 原文件
group_ids                {kb_id, node_id, txt}
status: released
```

每篇文档的数据散在三处，靠 `node_id` 串联。

## 易错典型

| 想法 | 实际 |
|---|---|
| "把 base64 文件塞 PG 字段" | ❌ 文件去 MinIO，PG 只存 URL |
| "把 session 存 PG" | ❌ 慢；用 Redis |
| "把向量存 PG（pgvector）" | ❌ 当前项目用 Qdrant，不要混 |
| "把热点 KB 配置查 N 次 PG" | ⚠️ 加 Redis 缓存 |

## 跨存储一致性

- PG 是权威源（source of truth）
- Redis 缓存策略：失效优先于更新
- MinIO 文件丢失：可重传（前提是有原始来源）
- Qdrant 向量丢失：触发 re-learning 即可恢复
