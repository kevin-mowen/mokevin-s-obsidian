# PandaWiki 启动流程总览（阶段1 MOC）

## 学习目标

读懂从 `go run ./cmd/api`（或 `./cmd/consumer`）到「HTTP 端口监听 / NATS 消费循环」之间发生了什么。重点掌握 [[Wire自注册模式]]：handler 构造函数本身就是「写路由表」的地方。

## 三个 cmd 进程对比

| 进程 | 入口 | 主体动作 | 端口 |
|---|---|---|---|
| API | `cmd/api/main.go` | `echo.Start(":18000")` 阻塞 HTTP 监听 | 18000 |
| Consumer | `cmd/consumer/main.go` | `MQConsumer.StartConsumerHandlers(ctx)` 阻塞 NATS 订阅 | 无 |
| Migrate | `cmd/migrate/main.go` | 运行一次性 SQL 迁移后退出 | 无 |

> 三者共用同一份 `setup`、`config`、`log`、`store`、`repo`、`usecase`，但选择性装入 handler。

## 启动两段式

每个进程都长这个结构：

```go
func main() {
    app, err := createApp()   // Wire 生成的依赖装配
    // ... 启动 echo / 消费循环 / 迁移
}
```

`createApp()` 由 `wire_gen.go` 自动生成（`go:build !wireinject`），它做的事就是按依赖拓扑序逐行 `New` 出每个对象，最后塞进一个 `*App` 结构体返回。`wire.go` 里那个 `wire.Build(...)` 在编译期失效（`//go:build wireinject`），只起「声明 ProviderSet」的作用。

## 关键洞察

1. **`*App` 里大部分字段是哑字段**。比如 `app.Handlers *v1.APIHandlers` 这个结构体在 `main()` 里从来不被读取。它存在的唯一目的是「逼 Wire 把所有 handler 实例化一次」；handler 在 `New*Handler` 里调用 `e.Group(...).GET/POST(...)` 把路由灌入 echo——*副作用就是装配本身*。
2. **`echo *echo.Echo` 在 Wire 图里被到处共享**。所有 handler 拿同一个 echo 实例，靠它注册路由。这意味着 echo 的初始化（`NewEcho`）必须先于所有 handler。
3. **API 和 Consumer 不是同一个进程**。它们是两个独立 binary，Wire 装配出两份 *App，靠 NATS（[[NATS]]）作为唯一通信通道。详见 [[Backend-API进程]] 和 [[Backend-Consumer进程]]。

## 自测题（阶段1结束前回答）

1. 为什么 `wire.go` 用 `//go:build wireinject` 标签而 `wire_gen.go` 用 `//go:build !wireinject`？
2. 给 API 加一个新的 `/api/v1/foo` 接口，需要改哪几个文件？
3. 给 Consumer 加一个新的 NATS 订阅 topic，需要改哪几个文件？
4. 假如 `NewUserHandler` 里忘了写 `e.Group("/api/v1/user")`，运行时会发生什么？
5. `*App` 里的 `Telemetry *telemetry.Client` 字段在 `cmd/api/main.go` 里没被显式调用，它怎么生效？
