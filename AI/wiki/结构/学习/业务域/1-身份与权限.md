# 业务域 1　身份与权限

> 决定"谁能访问什么"。多源认证 + 分组授权 + 多种 Token。

## 域内 domain 文件

| 文件 | 职责 |
|---|---|
| `auth.go` | JWT 用户登录、Session |
| `auth_group.go` | 用户组 / 部门树 |
| `api_token.go` | KB 级 API Token |
| `system_api_tokens`（migration 49/50） | 系统级 API Token，给对外 OpenAPI 用 |

## 主要 PG 表

```
users
auth_groups          (树形：parent_id 自引用)
kb_users             (kb_id × user_id 多对多)
api_tokens           (kb 级)
system_api_tokens    (系统级，带 secret 用于加密 access_credential)
```

## 三层鉴权

| 层 | 鉴权对象 | 关键中间件 |
|---|---|---|
| 系统级 | 是不是登录用户 | `middleware/auth.go`（JWT） |
| 知识库级 | 用户能否访问这个 kb | `kb_users` |
| 节点级 | 用户能否看这个 node | `auth_groups` × node.group_ids |

## 多源认证

- 本地账号（用户名+密码 / JWT）
- CAS SSO（公司内）→ 见 `usecase/auth_cas.go`
- GitHub OAuth → `usecase/auth_github.go`
- Pro 版的 LDAP / 飞书 / 钉钉等

## 核心约束

> **所有节点级权限检查都基于 `node.group_ids ⊂ user.group_ids`**

- Search 接口已有完整权限检查（`usecase/search.go` L444-497）
- ❌ Chat 接口缺少节点级权限检查 → 见 MEMORY"待修复问题：智能问答权限绕过"
- ❌ FAQ 节点必须继承父 node group_ids

## 三套 Token 不要混

| Token 类型 | 表 | 用途 | 头 |
|---|---|---|---|
| JWT | （Redis） | 登录 Session | `Authorization: Bearer <jwt>` |
| KB 级 API Token | `api_tokens` | 单个 kb 自动化 | `X-API-Token` |
| 系统级 API Token | `system_api_tokens` | 跨 kb / Skill / 第三方 | `Authorization: Bearer <token>` + secret |

## CUC 同步

公司组织架构通过 `cuc_sync` 配置定时从 `appstore.oa.cib:80` 拉取 → 写入 `auth_groups` 和 `users`。详见 MEMORY 的"CUC 用户同步性能优化"。
