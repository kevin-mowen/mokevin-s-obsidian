# 业务域 2　知识库与文档（核心）

> ⭐ **整个项目最核心的子域**。其他业务都绕着 kb 和 node 转。

## 域内 domain 文件

| 文件 | 大小 | 职责 |
|---|---|---|
| `knowledge_base.go` | 5.6k | 知识库（kb）创建/配置/成员 |
| `node.go` | 14k | 文档节点（CRUD + 树形 + 版本 + 发布） |
| `app.go` | 24k | 知识库的"前台门户配置"（域名、样式、机器人接入） |

## 三个核心概念

### 1️⃣ KB（Knowledge Base）

一组文档的集合，对应一个独立的 Wiki 站点。

```sql
knowledge_bases
  id, name, dataset_id (raglite 那边的 id), access_settings, ...
```

### 2️⃣ Node（文档节点）

文档树的节点。可以是文件夹、文档、外链。

```sql
nodes
  id, kb_id, parent_id (树形), type (folder/doc/link),
  name, content, group_ids (权限), status, version, ...
```

### 3️⃣ NodeRelease（已发布版本）

```
node 当前可编辑的版本（草稿）
   │
   │ 用户点击"发布"
   ▼
NodeRelease (快照) ──→ 触发 NATS → Consumer → raglite 学习
   │
   ▼
前台 App 看到的就是 NodeRelease，不是 node
```

## status 状态机

```
draft → pending(学习中) → released
                      └→ failed(学习失败)
```

学习失败的状态同步是已知 bug（见 MEMORY"RAG 链路问题清单"）。

## App（前台门户）

一个 kb 可以挂多个 App：
- web 网站门户
- widget 网页挂件
- bot：钉钉/飞书/企微/微信公众号/Discord
- API 机器人

每个 App 有独立的域名、样式、AI 配置。

## 数据流

```
用户传文档 → API → 写 nodes 表 → 发 NATS → Consumer → raglite 学习 → Qdrant
                                                                │
用户访问前台 App → API（handler/share）→ NodeRelease 表 ◄────────┘
```

## 关联表

```
node (主)
  ├─→ knowledge_base (所属 kb)
  ├─→ auth_groups (group_ids 多对多)
  ├─→ node_releases (历史发布快照)
  ├─→ comments / feedback (用户反馈)
  └─→ stat_pages (访问统计)
```
