# 五大子系统

> PandaWiki 整个项目按"职责"拆成五个互相协作的子系统。记住这五个，剩下的都是细节。

## 一图速览

```
①前端 ──→ ②后端业务 ──→ ③存储（PG/Redis/MinIO）
              │
              ├─→ ④消息（NATS）─→ Consumer 异步活
              │
              └─→ ⑤AI 链路（Raglite + Qdrant + Ollama）
```

## 五大子系统

| # | 子系统 | 进程位置 | 代表目录 | 关键认知 |
|---|---|---|---|---|
| **1** | 前端展示 | 宿主机 `pnpm dev` | `web/admin/` `web/app/` `web/packages/` | 两个独立应用：Admin 给管理员，App 给读者 |
| **2** | 后端业务 | 宿主机 `go run` | `backend/cmd/api/` `backend/cmd/consumer/` | 拆成两个进程：API 接 HTTP（要快），Consumer 啃异步活（可慢） |
| **3** | 数据存储 | Docker 容器 | PG / Redis / MinIO | 结构化用 PG，缓存用 Redis，文件用 MinIO |
| **4** | 异步消息 | Docker 容器 | NATS | 仅供 Backend 内部用，不对外暴露 |
| **5** | AI 链路 | Docker + 宿主机 Ollama | Raglite + Qdrant + Crawler + Ollama | PandaWiki 自己不做向量化，全部委托给 raglite |

## 为什么这么分

- **前后端拆**：前端要 SEO（App 用 Next.js SSR），不与后端耦合
- **API 与 Consumer 拆**：避免大文档学习卡死用户请求
- **AI 链路独立成 raglite 服务**：闭源算法 + 可独立扩容
- **基础设施容器化、业务代码本地跑**（dev 模式）：拿到热重载的同时锁定基础服务版本

## 易错点

- ❌ 把"AI 链路"和"后端"混为一谈：实际是 Backend → 调 raglite → raglite 才调 Ollama
- ❌ 以为 PG 存向量：向量在 [[Qdrant]]
- ❌ 以为 App 只服务登录用户：匿名也能访问公开知识库
