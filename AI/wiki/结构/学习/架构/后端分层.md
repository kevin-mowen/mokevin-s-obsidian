# 后端分层（Clean Architecture 风格）

> Backend 采用经典分层：handler → usecase → repo → domain，依赖单向。基础设施是 store、配置是 migration。

## 分层与职责

```
HTTP 请求
  ↓
handler/    ← 入口：路由、参数校验、HTTP 协议
  ↓
usecase/    ← 业务：组合多个 repo、事务、流程编排
  ↓
repo/       ← 数据访问：SQL/ORM、缓存读写
  ↓
domain/     ← 模型：纯 struct、DTO、常量（无业务逻辑）
  ↓
store/      ← 基础设施：pg / redis / mq 连接
  ↓
migration/  ← 数据库 schema（DDL）
```

## 各层规范

| 层 | 路径 | 职责 | 禁忌 |
|---|---|---|---|
| **domain** | `backend/domain/*.go` | 纯数据结构、CreateReq/UpdateReq、枚举 | ❌ 不能写业务逻辑 |
| **migration** | `backend/store/pg/migration/*.sql` | 建表、加字段、加索引；有 up/down 配对 | 编号递增，不可改已合并的 |
| **repo** | `backend/repo/pg/*.go` | CRUD 细节，只关心怎么存怎么查 | ❌ 不做业务判断 |
| **usecase** | `backend/usecase/*.go` | 业务规则、调多个 repo、组合数据 | ❌ 不直接写 SQL，不接 HTTP |
| **handler** | `backend/handler/{v1,share,openapi,mq}/*.go` | 绑路由、解参、调 usecase、返 JSON | ❌ 不写业务逻辑 |

## handler 子目录的含义

- `handler/v1/` — Admin 管理后台 API（`/api/v1/*`）
- `handler/share/` — App 前台 API（`/share/*`）
- `handler/openapi/` — 对外开放 API（`/openapi/*`）
- `handler/mq/` — 在 [[Backend Consumer 进程]] 里跑，订阅 NATS 消息
- `handler/pro/` — Pro 版本专属功能

## 依赖方向（单向）

```
handler → usecase → repo → domain
            ↓
         store（基础设施）
```

下层不感知上层。usecase 不知道 handler 的存在，repo 不知道 usecase 在干什么。这样 usecase 可以单独写测试（mock repo）。

## Wire 自动装配

每层定义 `Provider`，Wire 在编译期生成 `wire_gen.go`：

- 新加一个 handler/usecase/repo 后跑 `bash scripts/wire-auto-register.sh`
- 见 [[Wire-依赖注入多重绑定错误]] 处理冲突

## 易错点

- ❌ 在 domain 写方法 → 应该全部往 usecase 挪
- ❌ 在 handler 里写 SQL → 必须经由 repo
- ❌ usecase 调 handler → 永远反着的
