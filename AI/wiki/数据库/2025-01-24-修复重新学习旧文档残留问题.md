# 修复重新学习旧文档残留问题

**日期**: 2025-01-24
**修改文件**: `backend/handler/mq/rag.go`
**影响模块**: MQ Handler - RAG 向量化处理

## 问题描述

用户在执行"重新学习"操作后，前端仍然显示旧的学习失败错误信息，例如：

```
向量数据库更新失败: failed to update chunk xxx keywords: SetPayload() failed:
raglite: rpc error: code = NotFound desc = Not found: No point with id xxx found
```

## 问题分析

### 问题原因

重新学习时的代码流程存在缺陷：

1. 创建新文档到 raglite（获得新的 doc_id）
2. 更新 `node_releases.doc_id` 为新的 doc_id
3. **但从未删除旧文档** ← 问题所在

这导致：
- raglite 中存在多个版本的文档（旧的失败版本 + 新的成功版本）
- 旧的失败文档可能干扰状态同步
- 前端显示的是旧文档的失败状态

### 影响范围

- 文档重新学习功能
- RAG 向量化 MQ 处理流程
- 不影响首次学习、删除文档等其他操作

## 修改方案

### 修改内容

在 `HandleNodeContentVectorRequest` 函数的 `upsert` 操作中，添加在创建新文档前删除旧文档的逻辑。

### 代码改动

**文件**: `backend/handler/mq/rag.go`
**位置**: `upsert` case 中，`UpsertRecords` 调用之前（约第 95-108 行）

```go
// Delete old document if exists (for restudy scenario)
// This prevents multiple document versions from accumulating in raglite
if nodeRelease.DocID != "" {
    h.logger.Info("deleting old document before upsert",
        log.String("old_doc_id", nodeRelease.DocID),
        log.String("node_release_id", request.ID))
    if err := h.rag.DeleteRecords(ctx, kb.DatasetID, []string{nodeRelease.DocID}); err != nil {
        h.logger.Warn("failed to delete old document, continuing with upsert",
            log.String("old_doc_id", nodeRelease.DocID),
            log.Error(err))
        // Don't return error, continue with upsert
        // Old document may not exist or already deleted
    }
}
```

### 为什么这样改

1. **最小改动原则**: 只在必要位置添加删除逻辑，不修改其他代码
2. **容错设计**: 删除失败不阻断主流程，因为旧文档可能已不存在
3. **向后兼容**: 不改变接口签名，不影响其他调用方
4. **可追溯性**: 添加详细日志便于排查

## 改动说明

- **改动行数**: 约 14 行
- **影响模块**: MQ Handler - RAG 向量化处理
- **兼容性**: 向后兼容，不改变接口

## 验证方式

### 验证修复

1. 选择一篇学习失败的文档
2. 点击"重新学习"
3. 观察日志确认有 "deleting old document before upsert" 输出
4. 确认学习成功后，前端显示成功状态

### 回归测试

1. **首次学习**: 新文档首次学习应正常（DocID 为空，不触发删除）
2. **重新学习**: 已学习文档重新学习，旧文档应被删除
3. **删除文档**: 删除文档功能应正常工作
4. **学习失败重试**: 达到最大重试次数后应正确标记失败状态

### 日志检查

重新学习时应看到类似日志：

```
INFO  deleting old document before upsert  old_doc_id=xxx node_release_id=yyy
INFO  upsert node content vector success  node_release_id=yyy
```

## 补充修复：前端显示旧失败状态问题

**日期**: 2025-01-24
**修改文件**: `backend/usecase/node.go`

### 问题描述

用户点击"重新学习"后，虽然学习实际成功，但前端仍显示旧的失败错误信息。

### 问题分析

时序问题：
1. 用户点击"重新学习"
2. `NodeRestudy` 发送 MQ 消息（异步处理）
3. 前端立即开始轮询（每2秒）
4. 轮询从数据库获取状态，**此时状态还是旧的 `ENHANCE_FAILED`**
5. 前端看到 `ENHANCE_FAILED`（终态），认为学习完成，显示失败
6. 实际上 MQ 还没处理完，处理完后状态才会更新

**根本原因**：`NodeRestudy` 没有立即更新节点状态为"学习中"，导致前端轮询时读到旧的失败状态。

### 修复内容

在 `NodeRestudy` 函数中，发送 MQ 消息前先将节点状态更新为 `BASIC_RUNNING`：

```go
// 立即更新节点状态为"学习中"，防止前端轮询时读到旧的失败状态
if err := u.nodeRepo.Update(ctx, nodeRelease.NodeID, map[string]any{
    "rag_info": domain.RagInfo{
        Status:  consts.NodeRagStatusBasicRunning,
        Message: "",
    },
}); err != nil {
    u.logger.Error("update node rag status to running failed",
        log.String("node_id", nodeRelease.NodeID),
        log.Error(err))
    // 即使更新失败也继续发送 MQ 消息
}
```

### 改动说明

- **改动行数**: 约 10 行
- **影响模块**: NodeRestudy 用例
- **兼容性**: 向后兼容

### 修复后的流程

1. 用户点击"重新学习"
2. `NodeRestudy` **立即**将状态更新为 `BASIC_RUNNING`
3. 前端开始轮询，看到 `BASIC_RUNNING`（非终态），继续轮询
4. MQ 处理完成，状态更新为 `BASIC_SUCCEEDED`
5. raglite 增强处理完成，状态同步更新为 `ENHANCE_SUCCEEDED`
6. 前端看到 `ENHANCE_SUCCEEDED`，显示成功
