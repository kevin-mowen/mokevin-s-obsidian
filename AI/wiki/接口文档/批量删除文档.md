# 批量删除文档（仅管理员）

## 接口信息

| 项目               | 值                                          |
| ---------------- | ------------------------------------------ |
| **路径**           | `POST /api/v1/kb/doc/delete`               |
| **认证**           | `Authorization: Bearer <system_api_token>` |
| **Content-Type** | `application/json`                         |
| **权限要求**         | 系统级 API Token + **知识库 full_control 管理员**   |

## 功能说明

一次性删除一个知识库内的多篇文档（最多 20 个）。调用前会逐个校验每个 doc_id：存在 / 属于本 KB / 类型为 document；不通过的条目进入 `failed` 列表返回，其余有效 id 走统一的删除动作。**仅知识库 full_control 管理员可调用**，比创建/编辑接口更严格。

## 请求参数

| 参数                | 类型       | 必填 | 说明                           |
| ----------------- | -------- | -- | ---------------------------- |
| system_id         | string   | 是  | 调用方系统标识                      |
| agent_id          | string   | 否  | 智能体 ID                       |
| access_credential | string   | 是  | AES-256-GCM 加密后的用户工号（base64） |
| kb_id             | string   | 是  | 知识库 ID                       |
| doc_ids           | string[] | 是  | 要删除的文档 ID 列表，长度 1-20         |

### doc_ids 规则

- 不能为空数组
- 单次最多 20 个，超过返回 40001
- 每个 id 会独立校验：不存在 / 跨 KB / 是文件夹 都进入 `failed`，不影响其它 id

## curl 示例

```bash
curl -X POST 'http://dev.localhost:18000/api/v1/kb/doc/delete' \
  -H 'Authorization: Bearer <system_api_token>' \
  -H 'Content-Type: application/json' \
  -d '{
    "system_id": "obsidian",
    "access_credential": "AES-256-GCM 加密后的用户工号（base64）",
    "kb_id": "be8666e3-7823-4e20-9f1c-f77ac604b47a",
    "doc_ids": [
      "019b4a7e-793b-7743-857a-632bf37e3b12",
      "019b4a7e-792d-754f-bbe3-9c50d2622603"
    ]
  }'
```

## 响应示例

### 全部成功

```json
{
  "success": true,
  "code": 0,
  "message": "success",
  "data": {
    "deleted": [
      "019b4a7e-793b-7743-857a-632bf37e3b12",
      "019b4a7e-792d-754f-bbe3-9c50d2622603"
    ],
    "failed": []
  }
}
```

### 部分失败（成功删除可删的部分，失败项逐条说明）

```json
{
  "success": true,
  "code": 0,
  "message": "success",
  "data": {
    "deleted": ["019b4a7e-793b-7743-857a-632bf37e3b12"],
    "failed": [
      { "doc_id": "00000000-0000-0000-0000-000000000000", "reason": "文档不存在" },
      { "doc_id": "019db082-2629-7115-b066-2e92f6036485", "reason": "该节点是文件夹，请使用 /folder/delete" }
    ]
  }
}
```

### 全部不可删（无有效 id，返回 40003）

```json
{
  "success": false,
  "code": 40003,
  "message": "没有可删除的文档",
  "data": {
    "deleted": [],
    "failed": [
      { "doc_id": "xxx", "reason": "文档不属于该知识库" }
    ]
  }
}
```

## 响应字段

| 字段                | 类型       | 说明                             |
| ----------------- | -------- | ------------------------------ |
| deleted           | string[] | 本次实际删除的文档 id 列表                |
| failed            | array    | 校验未通过未删除的条目                    |
| failed[].doc_id   | string   | 未删除的文档 id                      |
| failed[].reason   | string   | 失败原因：文档不存在 / 文档不属于该知识库 / 该节点是文件夹 |

## 权限检查流程

1. **系统级 Token 校验** — 必须使用系统级 API Token（KB 级别 Token 不可用）
2. **基础参数校验** — system_id / access_credential / kb_id / doc_ids 都不能为空，doc_ids ≤ 20
3. **access_credential 解密** — 用 Token 的 secret 做 AES-256-GCM 解密得到 notes_id
4. **用户身份解析** — 通过 notes_id 查 authID，不存在返回 40003
5. **知识库授权** — kb_id 必须在 Token 授权的知识库范围内
6. **管理员严格校验** — notes_id 对应的 admin user 在 `kb_users` 中必须是 `full_control`，否则返回 40003

## 错误码

| code  | 说明                                             |
| ----- | ---------------------------------------------- |
| 40001 | 参数错误（缺字段 / doc_ids 为空 / doc_ids > 20 / 解密失败）  |
| 40003 | 权限不足（用户不存在 / 无 KB 权限 / 非 full_control / 无可删文档）|
| 50001 | 查询文档信息失败                                       |

其它删除阶段异常统一走 `NewResponseWithError`。

## 注意事项

- **单次上限 20**：超过请客户端分批；同一 KB 内循环调用即可。
- **权限级别高于创建**：创建文件夹/文档允许 `doc_manage`，删除只允许 `full_control`，所以有 "能建但不能删" 的账号存在。
- **类型隔离**：这个接口**不**处理文件夹，传入文件夹 id 会进 `failed`，不会误删整棵子树；要删文件夹走 `/api/v1/kb/folder/delete`。
- **跨 KB 保护**：doc_id 属于别的 KB 会被拦在 `failed`，不会越权删除。
- **原子性**：本接口按"预校验 + 统一 NodeAction"两段执行，预校验失败的条目不会进入删除动作；有效 id 一起进入底层 delete action。
