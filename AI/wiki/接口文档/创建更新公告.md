# 创建更新公告

## 功能描述

通过系统级 API Token 创建一条门户更新公告，对应管理后台「公告」菜单的新建公告功能的接口化版本。公告可选择「立即发布」或保持「未发布」状态；未发布公告仅 admin 后台可见，发布后门户首页弹窗对所有用户展示。

## 功能特性

- 仅接受**系统级 API Token**，KB 级 Token 与 Cookie 鉴权均会被拒绝
- 支持 6 种条目类型（feature / improve / fix / breaking / security / maint），前端弹窗按类型显示不同颜色标签
- 创建后可选择是否立即发布（`is_published`）
- 与 admin 后台共用同一份 `usecase` 与数据表，DB 数据互通
- 创建后若系统配置了 `announcement_target`，自动重新渲染所有已发布公告并同步到指定 KB Node

## 实现文件

### 后端

- `backend/handler/openapi/announcement.go` — 路由注册 `POST /api/v1/kb/announcement/create` 与 `Create` handler
- `backend/handler/openapi/auth_helper.go` — 包级 `requireSystemToken` 校验函数（与 `kb.go` 共用）
- `backend/handler/openapi/provider.go` — wire ProviderSet 注册 `OpenAPIAnnouncementHandler`
- `backend/usecase/announcement.go` — `AnnouncementUsecase.Create` 业务逻辑（本接口复用，admin 端共用）
- `backend/domain/announcement.go` — `AnnouncementCreateReq` / `AnnouncementItem` / `AnnouncementItemType` 枚举
- `backend/repo/pg/announcement.go` — 数据访问层

### 前端

- 管理后台「公告」菜单：`web/admin/src/pages/announcement/index.tsx` / `Form.tsx`
- 门户首页公告弹窗：`web/app/src/components/AnnouncementModal/index.tsx`

> 注：本接口不直接驱动前端，前端通过 admin 端接口（`/api/v1/announcement` 系列）操作。

## API 接口

| 方法 | 路径 | Content-Type |
|------|------|--------------|
| POST | `/api/v1/kb/announcement/create` | `application/json` |

> 路径挂在 `/api/v1/kb/*` 命名空间下是为了匹配生产反向代理已放行的白名单，**资源本身仍是系统级**（不绑定到任何 KB）。

## 请求头

| Header | 必填 | 说明 |
|--------|------|------|
| Authorization | 是 | `Bearer <system_api_token>`，仅支持系统级 API Token，KB 级 Token 不可用 |

## 数据结构

### 请求字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| system_id | string | 是 | 调用方系统标识（仅审计用） |
| agent_id | string | 否 | 智能体 ID（仅审计用） |
| version | string | 是 | 版本号，DB 唯一索引，重复会失败 |
| title | string | 是 | 公告标题，弹窗顶部展示 |
| publish_date | string | 是 | 发布日期，格式必须为 `YYYY-MM-DD` |
| items | array | 是 | 更新条目数组，至少 1 条 |
| items[].type | string | 是 | 条目类型，6 种枚举（见下） |
| items[].title | string | 是 | 条目标题 |
| items[].summary | string | 否 | 条目摘要，可省略 |
| is_published | bool | 否 | 是否立即发布，默认 `false` |

### items[].type 枚举

| 值 | 中文标签 | 弹窗颜色 |
|----|---------|---------|
| feature | 新功能 | 蓝 #3B6EF5 |
| improve | 优化 | 绿 #15B392 |
| fix | 修复 | 紫 #8B5CF6 |
| breaking | 重要变更 | 红 #E5484D |
| security | 安全通告 | 橙 #D97706 |
| maint | 停机维护 | 灰 #64748B |

### 响应字段

| 字段 | 类型 | 说明 |
|------|------|------|
| data | string | 新建公告的 UUID（可在 admin 后台按此 ID 查找） |

## 数据结构（DB）

### 主表: announcements

| 字段 | 类型 | 说明 |
|-----|------|------|
| id | string (uuid) | 主键，UUID v7 |
| version | string | 版本号，唯一索引 |
| title | string | 标题 |
| publish_date | date | 发布日期 |
| items | jsonb | 条目列表（数组） |
| is_published | bool | 是否已发布，默认 false |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

索引: `version` 唯一索引

## curl 示例

```bash
curl -X POST 'http://dev.localhost:18000/api/v1/kb/announcement/create' \
  -H 'Authorization: Bearer <system_api_token>' \
  -H 'Content-Type: application/json' \
  -d '{
    "system_id": "manual-publish",
    "agent_id": "claude-code",
    "version": "v1.9.0",
    "title": "导入入口精简 + 大文档性能升级",
    "publish_date": "2026-04-29",
    "items": [
      {"type": "feature", "title": "新增门户更新公告模块", "summary": "后台创建/编辑/发布版本公告"},
      {"type": "improve", "title": "大 Markdown 渲染性能大幅提升"},
      {"type": "fix", "title": "修复本地文件导入图片渲染异常"}
    ],
    "is_published": false
  }'
```

## 权限控制

| 操作 | 所需权限 |
|------|---------|
| 调用本接口 | 系统级 API Token + Token 权限为 `full_control` 或 `doc_manage` |
| KB 关联 | 无 — 公告是系统级数据，不绑定到具体 KB（路径在 `/api/v1/kb/*` 下仅是为了过反代白名单） |
| 用户身份 | 无 — 公告不归属某个真实用户 |

### 校验流程

1. 系统级 Token 校验（KB 级 Token 与 Cookie 鉴权拒绝）
2. Token 权限为 `full_control` 或 `doc_manage`
3. 请求参数必填校验（system_id / version / title / publish_date / items）
4. 调用 `AnnouncementUsecase.Create` 落库
5. 触发 `trySyncToNode`（若系统配置了 `announcement_target`）
6. 写入 OpenAPI 调用日志

## 依赖关系

- **上游依赖**：系统 API Token 模块、`AnnouncementUsecase`、`AnnouncementRepository`、`SystemSettingRepo`（`announcement_target` 配置）、`NodeUsecase`（公告同步到 KB Node）、`KnowledgeBaseUsecase`（KB Release）
- **下游影响**：
  - admin 后台「公告」菜单（共享同一张表，新建公告会出现在列表中）
  - 门户首页公告弹窗（`is_published=true` 后立即可见）
  - 若配置了同步目标 KB Node，对应 Node 内容会被覆盖

## 功能边界

- **本模块负责**：通过系统级 API Token 创建公告、参数校验、写入调用日志
- **本模块不负责**：
  - 公告列表查询（由 admin 接口 `/api/v1/announcement/list` 处理）
  - 公告详情查询（由 admin 接口 `/api/v1/announcement/{id}` 处理）
  - 公告更新（由 admin 接口 `PUT /api/v1/announcement/{id}` 处理）
  - 公告删除（由 admin 接口 `DELETE /api/v1/announcement/{id}` 处理）
  - 切换发布状态（由 admin 接口 `POST /api/v1/announcement/{id}/publish` 处理）
  - 同步目标 KB Node 配置（由 admin 接口 `/api/v1/announcement/config` 处理）
  - 门户读取已发布公告（由 share 接口 `backend/handler/share/announcement.go` 处理）
  - 公告渲染为 Markdown / HTML 与 KB Release（由 `usecase.AnnouncementUsecase.trySyncToNode` 处理）

## 注意事项

- `version` 必须全局唯一，重复传入会被 DB 唯一索引拒绝（错误信息中包含 trace_id，日志层定位具体原因）
- `publish_date` 严格 `YYYY-MM-DD`，其他格式（如 `2026/04/29`、`2026-4-29`）会失败
- `items` 不能为空数组；条目内 `type` 不在枚举列表内不会报错，但前端弹窗会回退为原始字符串展示
- `is_published=false` 时公告已落库但门户用户看不到，需到 admin 后台或调 admin 端 publish 接口手动发布
- 接口无更新/删除能力，建错请通过 admin 后台或 admin 端接口处理
- 路径前缀 `/api/v1/kb/*` 仅是反向代理白名单适配；不会与 kb.go 已有路由（`/list`、`/folders`、`/upload` 等）冲突，子路径 `announcement/create` 唯一

## 错误码

| code | HTTP | 说明 |
|------|------|------|
| 0 | 401 | 未带 Token 或 Token 无效 |
| 401 | 401 | KB 级 Token 调用（被 `requireSystemToken` 拦截） |
| 403 | 403 | Token 权限不足（需要 `full_control` 或 `doc_manage`） |
| 40001 | 200 | 参数错误（缺字段或请求体 JSON 解析失败） |
| 0 | 200 | 创建公告失败（含 trace_id），常见于 publish_date 格式错误或 version 重复 |

## 调用日志

成功创建后写入 `openapi_call_logs` 表（失败不写）：

| 字段 | 取值 |
|------|------|
| system_id | 请求体 `system_id` |
| agent_id | 请求体 `agent_id` |
| api_token_id | 系统级 Token 对应的 user_id |
| endpoint | `/api/v1/kb/announcement/create` |
| remote_ip | `c.RealIP()` |
| success | true |

## 踩坑记录

### 问题 1: 提取 `requireSystemToken` 为包级函数时影响了 kb.go 14 个接口

- 原因：`requireSystemToken` 原本是 `OpenAPIKBHandler` 的私有方法（receiver 写法），但函数体未使用 `h`，是历史代码挂载选择
- 解决：将其提取为 `auth_helper.go` 中的包级函数，kb.go 内 15 处 `h.requireSystemToken(c)` 调用全部去掉 `h.` 前缀；运行时行为完全等价（已逐个回归测试）

### 问题 2: GORM `bool` 字段零值陷阱（项目既有问题）

- 原因：`OpenAPICallLog.Success` 字段定义为 `bool` + `default:true`，GORM 默认忽略零值（`false`）字段，导致 `Success: false` 写入时使用 DB 默认值 `true`
- 影响：openapi 包内所有"失败时也写日志"的实现都会被错误标记为成功
- 解决：本接口与 kb.go 一致**只在成功路径写日志**，回避此陷阱；如需修，需把 model 字段改成 `*bool` 或调整 default

### 问题 3: 生产反向代理只放行 `/api/v1/kb/*`，原路径 `/api/v1/announcement/create` 被 SPA 兜底吞掉

- 原因：生产 caddy/nginx 配置只 path-prefix 转发 `/api/v1/kb/*` 到 backend；其他 `/api/v1/*` 全部走前端 SPA index.html 兜底，请求根本到不了 Echo router。表现为外部 token 调用 admin 老接口（`/api/v1/announcement/list`）和我们新接口（`/api/v1/announcement/create`）都返回 34KB 的 HTML 404，与"完全不存在的路径"响应一致；而 `/api/v1/kb/ping`、`/api/v1/kb/list` 全部正常。Admin 后台公告功能在生产能用，是因为它走另一条内网通道、不经此外部反向代理
- 解决：把 OpenAPI 公告接口路径迁移到 `/api/v1/kb/announcement/create`，命中现有反向代理白名单。语义上公告**仍是系统级资源**，路径前缀只是基础设施适配；admin 端老接口路径保持不变（admin 走内网通道无影响）
- 长期建议：让运维把反向代理改为整段透传 `/api/v1/* → backend`，未来新增对外接口无需再迁就路径前缀；本次未走此路径是因为依赖外部团队配合
