## 二、核心功能流程

### 1. AI 问答流程（最核心、最复杂）

```
用户提问 → POST /share/v1/chat/message (SSE流式)
  │
  ├─ [1] 验证码校验
  ├─ [2] 获取/创建 App + 加载聊天模型配置
  ├─ [3] 创建/验证会话 (conversation_id + nonce)
  ├─ [4] 敏感词过滤 (DFA算法检测屏蔽词)
  ├─ [5] 获取用户权限组 (auth_groups)
  ├─ [6] RAG 检索 ← 关键步骤
  │     ├─ 加载历史对话
  │     ├─ 调用 Raglite 向量搜索 (带 group_ids 过滤)
  │     ├─ FilterRankedNodesByPermission() ← 按 "Answerable" 权限过滤
  │     └─ 格式化文档上下文 + 系统提示词
  ├─ [7] LLM 流式推理 (OpenAI/DeepSeek/Gemini/Ollama)
  │     └─ 每个 chunk 再做敏感词过滤
  ├─ [8] 保存消息到 conversation_messages
  └─ [9] 更新模型用量统计
```

**关键文件**：

| 文件 | 职责 |
|------|------|
| `backend/usecase/chat.go` | Chat() 方法 (L77-307)，问答主流程 |
| `backend/usecase/llm.go` | BuildConversationMessageWithRAG() (L63-149)，RAG + LLM 编排 |
| `backend/handler/share/chat.go` | HTTP 入口 |
| `backend/store/rag/ct.go` | Raglite SDK 调用 |
| `web/app/src/components/aiQaModal/index.tsx` | 前端 SSE 客户端 |
| `web/app/src/views/chat/ChatResult.tsx` | 对话渲染 + 反馈 |

### 2. 文档管理与发布流程

```
创建文档 → POST /api/v1/node
  │
编辑文档 → PUT /api/v1/node/detail (TipTap 富文本编辑器)
  │
发布文档 → POST /api/v1/node/action (action="publish")
  ├─ 创建 node_release 记录 (不可变快照)
  ├─ 设置 nodes.is_published = true
  └─ 发送 NATS 消息 → Consumer 异步索引到 RAG
       └─ Raglite.UpsertRecords(content, groupIDs)
           └─ 存入 Qdrant 向量数据库

删除文档 → POST /api/v1/node/action (action="delete")
  ├─ 获取关联的 doc_ids
  ├─ PostgreSQL 删除 (软/硬)
  └─ 发送 NATS 消息 → Consumer 异步从 RAG 删除
```

**关键文件**：

| 文件 | 职责 |
|------|------|
| `backend/usecase/node.go` | Node CRUD + 发布/删除逻辑 |
| `backend/repo/pg/node.go` | 数据库操作（最大的 repo 文件） |
| `backend/repo/mq/rag.go` | 异步 RAG 同步消息发送 |
| `web/admin/src/pages/document/` | 文档管理 UI（树形拖拽、批量操作） |
| `web/admin/src/pages/document/editor/` | TipTap 富文本编辑器 |

### 3. 搜索流程

```
用户搜索 → POST /share/v1/chat/search
  ├─ 获取用户权限组
  ├─ 调用 RAG 向量搜索
  ├─ FilterRankedNodesByPermission() ← 按 "Visitable" 权限过滤
  └─ 返回可见的文档结果
```

**与问答的区别**：搜索用 `NodePermNameVisitable`（可查看），问答用 `NodePermNameAnswerable`（可回答）

**关键文件**：`backend/usecase/chat.go` 的 `Search()` 方法 (L444-522)

### 4. 认证与权限模型

```
权限层级：
  ├─ 用户角色: Admin / User
  ├─ KB 权限: Null → DocManage → DocPublish → FullControl
  ├─ 认证组: 按来源分组 (LDAP, OAuth, CAS 等)
  └─ 文档级权限 (per auth_group):
       ├─ Visible   — 目录中可见
       ├─ Visitable — 可查看内容
       └─ Answerable — 可被 AI 引用回答

中间件链：
  Request → ReadOnly → Session → JWT → UserRole → KBPerm → Handler
```

**认证源**：本地密码、CAS SSO、微信、钉钉、企业微信

**关键文件**：

| 文件 | 职责 |
|------|------|
| `backend/middleware/jwt.go` | JWT / API Token 验证 |
| `backend/middleware/share_auth.go` | 公开端点认证 |
| `backend/usecase/auth.go` | 认证业务逻辑 (509行) |
| `backend/domain/auth.go` | 权限数据模型 |

## 八、已知风险与排查要点

### 1. Chat 权限绕过（高优先级）

- **问题**：`/share/v1/chat/completions` 端点缺少 `Authorize` 中间件
- **影响**：用户可能通过 AI 问答获取无权访问的文档内容
- **对比**：Search 方法有完整权限检查，Chat 的 `FilterRankedNodesByPermission` 存在但 `/completions` 入口未强制认证
- **相关文件**：`backend/handler/share/chat.go:67`、`backend/usecase/chat.go`

### 2. RAG 权限依赖应用层

- Raglite 查询时 group_ids 为空不会报错，返回所有结果
- 权限过滤完全依赖 `FilterRankedNodesByPermission()` 应用层代码
- 如果该函数有 bug，权限形同虚设
- **相关文件**：`backend/usecase/llm.go:459-518`、`backend/store/rag/ct/rag.go:49-52`

### 3. Raglite v2.x BaseURL 注意

- 默认端口从 `:8080/api/v1` 改为 `:5050`
- 地址**不能**带 `/api/v1` 后缀，新 SDK 会自动拼接路径
- 生产环境需通过 `RAGLITE_URL` 环境变量指定正确地址

## 九、问题排查快速定位

| 问题类型 | 排查路径 |
|---------|---------|
| AI 问答异常 | `handler/share/chat.go` → `usecase/chat.go` → `usecase/llm.go` → `store/rag/ct.go` |
| 文档不可见 | `middleware/jwt.go` 认证 → `repo/pg/node.go` 权限查询 → `domain/node.go` 权限模型 |
| RAG 不同步 | `repo/mq/rag.go` 消息发送 → NATS 日志 → Consumer `RAGMQHandler` → Raglite API |
| 搜索无结果 | `usecase/chat.go:Search()` → Raglite 向量检索 → Qdrant 索引状态 |
| 前端 Admin 报错 | `web/admin/src/request/httpClient.ts` 拦截器 → Redux store → 对应页面组件 |
| 前端 App 报错 | `web/app/src/provider/index.tsx` 上下文 → `web/app/src/request/httpClient.ts` |
| 认证失败 | `middleware/jwt.go` → `usecase/auth.go` → 对应 OAuth 源配置 |
| 消息队列积压 | NATS JetStream 监控 → Consumer 日志 → 对应 Handler 处理逻辑 |
| 文件上传失败 | `handler/v1/file.go` → MinIO 连接 → S3 配置 (`config.local.yml`) |
