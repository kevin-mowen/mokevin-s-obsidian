# 大 Markdown 文件编辑器性能优化分析

> 日期：2026-04-16
> 状态：方案已实现，部分已提交

## 问题描述

Admin 后台导入大型 .md 文件（600KB+、19000+ 行）后，打开文档时编辑器卡顿甚至崩溃，前台查看同样卡顿。

## 问题根因

### 编辑器架构

```
前端编辑器架构：
  Tiptap（封装层）→ ProseMirror（核心引擎）→ DOM 渲染

Markdown 模式加载流程：
  API 返回完整内容 → useTiptap 初始化 → Tiptap Markdown 解析器
  → 生成 ProseMirror 文档树（每个元素 = 一个节点）→ 全量渲染 DOM
```

### 卡顿/崩溃的真正原因

**不是文件大小本身，而是 ProseMirror 节点数量过多。**

每行 Markdown 至少生成 1 个 ProseMirror 节点，每个 HTML 标签额外生成节点。当节点数超过约 3000 个时：

| 问题 | 表现 |
|------|------|
| Markdown 模式 | Tiptap 解析崩溃：`Invalid content for node doc` |
| HTML 富文本模式 | 加载慢但不崩溃，功能正常 |

### 导致节点过多的内容特征

| 特征 | 举例 | 节点膨胀 |
|------|------|---------|
| 大量 HTML 表格 | `<table><tr><td>...` | 1 个 10x5 表格 = 61 个节点 |
| 内容行数极多 | 19000 行配置手册 | 每行至少 1 个节点 |
| `$` 符号 | `$n$`、`$1` | 被 Tiptap 数学扩展误识别为 LaTeX |
| HTML + Markdown 深度混合 | `### 标题` + `<table>` + `<img>` | 解析器模式切换出错 |

### 实际问题文件分析

测试文件：`01-基础配置指导.md.md`（652KB，19547 行）

- 文件后缀是 `.md`，但内容 100% 是 HTML（`<h3>`、`<p>`、`<table>` 等标签）
- 无 base64 图片（图片已是 MinIO URL）
- HTML 标签约 9800 个
- 通过 anydoc 服务处理后返回内容，前端以 `content_type=md` 存库

## 代码路径分析

### 前端加载路径

```
路由 /doc/editor/:id
  → Edit 组件 (editor/edit/index.tsx)
  → GET /api/v1/node/detail?format=raw
  → Handler (handler/v1/node.go:142)
  → Usecase (usecase/node.go:120) GetNodeByKBID()
  → Repo (repo/pg/node.go:329) GetByID() → SQL 查询
  → 返回 NodeDetailResp { Content: "完整内容" }
  → Wrap 组件 (editor/edit/Wrap.tsx:234)
  → useTiptap({ content, contentType }) → ProseMirror 解析
  → 根据 isMarkdown 渲染：
      isMarkdown=true  → EditorMarkdown (Ace Editor) + Tiptap 后台解析
      isMarkdown=false → FullTextEditor (ProseMirror 富文本)
```

### 导入路径

```
上传 .md 文件 → anydoc 服务处理（下载图片、转换格式）
  → ScrapeGetResult 返回内容
  → 前端判断 fileType=md → content_type: 'md'
  → POST /api/v1/node → Create() 存库
```

### 关键判断逻辑

- `IsLikelyHTML()`：检查内容是否以 `<` 开头且以 `>` 结尾
- `isMarkdown`（前端）：`content_type === 'md'`
- 编辑器选择：isExcel → ExcelEditor / isMarkdown → EditorMarkdown / 默认 → FullTextEditor

## 优化方案

### 方案 1：后端入库时转 HTML（推荐，已实现）

**原理**：在 `node.go` Create 方法中，检测 `content_type=md` 的内容节点数，超标时调用已有的 `ConvertMDToHTML` 转为 HTML 存储。

**修改文件**：
- `backend/usecase/node.go` — Create 方法新增两层检测
- `backend/utils/utils.go` — 新增 `ShouldConvertMDToHTML` 函数

**检测逻辑**：
```
Create 方法接收 content_type=md 的内容
  ↓
第一层：IsLikelyHTML — 内容是纯 HTML？
  → 是：清除 content_type，直接走富文本（不转换内容）
  ↓
第二层：ShouldConvertMDToHTML — 估算节点数 > 3000？
  → 是：ConvertMDToHTML 转为 HTML，清除 content_type
  → 否：保持 md，走 Markdown 编辑器
```

**节点数估算**：
```
estimatedNodes = 内容行数 + HTML 开标签数量
阈值 = 3000
```

**测试结果**：

| 内容 | 估算节点 | 是否转换 |
|------|---------|---------|
| 100 行普通 README | ~100 | 不转换 |
| 1200 行 + 200 个 HTML 标签 | ~1400 | 不转换 |
| 3000 行 + 2500 个 HTML 标签 | ~5500 | 转换 |
| 19000 行纯 HTML 文件 | ~29000 | 转换 |

**优点**：
- 改动最小（只改后端 2 个文件）
- 功能完整（富文本编辑器所有功能正常）
- 用户体验一致（不会出现两种编辑器 UI）
- 前端完全不用改

**缺点**：
- 转换后用户看到富文本编辑器，不再是 Markdown 源码编辑器
- HTML 比 Markdown 体积大 10-15%
- 富文本模式下大文件仍然加载慢（但不崩溃）
- 只解决新导入的文档，历史数据不处理

### 方案 2：前台 react-markdown 渲染（兜底历史数据，已实现）

**原理**：前台查看时，对大 Markdown 文件跳过 Tiptap，用已有的 `react-markdown` 组件直接渲染。

**修改文件**：
- `web/app/src/views/node/index.tsx` — 检测大 Markdown，跳过 Tiptap 初始化，正则提取 TOC
- `web/app/src/views/node/DocContent.tsx` — 新增 `markdownContent` prop，有值时用 MarkDown 组件渲染

**优点**：
- 秒开，性能最优
- 能处理历史已存储的大 Markdown 数据

**缺点**：
- 只解决前台查看，Admin 端编辑不处理
- 功能降级（无富文本工具栏）

### 方案 3：Admin 端独立编辑器（已实现后回退，边框样式待修复）

**原理**：Admin 编辑时，对大 Markdown 文件使用独立的 Ace Editor + react-markdown 预览，完全绕过 Tiptap。

**修改文件**：
- `web/admin/src/pages/document/editor/edit/Wrap.tsx` — 大文件检测、正则 TOC、条件渲染
- `web/admin/src/pages/document/editor/edit/LargeMarkdownEditor.tsx` — 新建独立编辑器组件
- `web/admin/src/pages/document/editor/edit/Toc.tsx` — TOC 点击定位 fallback

**当前状态**：已回退，原因是预览模式的边框样式问题未解决。

## 三个方案的关系

```
新导入的文档 → 方案 1（后端转 HTML，前端无感）
历史已存储的大 Markdown：
  前台查看 → 方案 2（react-markdown 兜底）
  Admin 编辑 → 方案 3（待修复边框样式后恢复）
```

## 排除的优化方向

| 方向 | 可行性 | 原因 |
|------|--------|------|
| 懒加载/虚拟滚动 | 低 | ProseMirror 需要完整文档树 |
| Web Worker | 低 | ProseMirror 强依赖 DOM |
| 精简 Extension | 中 | `@ctzhian/tiptap` 是外部包，扩展在包内部 |
| 存 JSON 格式 | 低 | 需改数据库和所有读写接口 |
| 分块编辑（Notion 式） | 低 | 架构级重构 |
| 换编辑器引擎 | 低 | 项目深度绑定 Tiptap |

## 其他发现

### anydoc 处理慢的问题

纯 HTML 内容的 .md 文件导入时，anydoc 服务会尝试下载内容中所有 `<img>` 标签的图片。如果图片地址不可达（如生产环境内网地址），每张图片超时 1-2 秒，累计导致导入耗时数分钟。

这是 anydoc 服务（`chaitin/anydoc:v0.5.7`）的行为，属于外部 Docker 镜像，不在项目代码控制范围内。

### ConvertMDToHTML 不会丢失内容

使用 `gomarkdown` 库 + `bluemonday` 安全过滤：
- Markdown 语法 → 对应 HTML 标签（不丢失）
- 内嵌 HTML 标签 → 原样保留（不丢失）
- `<script>`/`<iframe>` 等危险标签 → 被过滤（安全策略，知识库文档中不应出现）
