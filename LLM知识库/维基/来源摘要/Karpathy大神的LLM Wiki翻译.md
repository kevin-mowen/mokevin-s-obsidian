# LLM Wiki — 用 LLM 构建个人知识库的模式

一种使用大语言模型构建个人知识库的模式。

这是一份理念文档，设计初衷是供你复制粘贴到自己的 LLM Agent（如 OpenAI Codex、Claude Code、OpenCode / Pi 等）中使用。它旨在传达核心理念，而你的 Agent 会与你协作完成具体的实施细节。

## 核心理念

大多数人使用 LLM 处理文档的体验就像 RAG（检索增强生成）：你上传一批文件，LLM 在查询时检索相关片段，然后生成答案。这种方式可以用，但 LLM 每次都在**从零开始重新发现知识**。没有任何积累。你问一个需要综合五篇文档才能回答的精妙问题，LLM 每次都得重新找到并拼凑那些相关片段。什么都没有被建构起来。NotebookLM、ChatGPT 文件上传、以及大多数 RAG 系统都是这样运作的。

这里的理念完全不同。LLM 不再只是在查询时从原始文档中检索，而是**渐进式地构建和维护一个持久化的 Wiki** — 一组结构化的、相互链接的 Markdown 文件，处于你和原始资料之间。当你添加一个新来源时，LLM 不只是把它索引起来留待日后检索，而是会阅读它、提取关键信息、并将其**整合到现有的 Wiki 中** — 更新实体页面、修订主题摘要、标注新数据与旧结论的矛盾之处、强化或挑战正在演进中的综合分析。知识被编译一次，然后**持续保持更新**，而不是每次查询时重新推导。

这就是关键区别：**Wiki 是一个持久的、不断复合增值的制品。** 交叉引用已经建好了，矛盾已经被标记了，综合分析已经反映了你读过的所有内容。每添加一个来源、每提出一个问题，Wiki 就会变得更加丰富。

你几乎不需要（或很少需要）亲自写 Wiki — LLM 负责编写和维护所有内容。**你负责的是寻找资料来源、探索方向、提出正确的问题。** LLM 做所有苦力活 — 总结、交叉引用、归档、以及让知识库随时间变得真正有用的簿记工作。在实际操作中，我一边开着 LLM Agent，一边开着 Obsidian。LLM 基于我们的对话进行编辑，我实时浏览结果 — 跟踪链接、查看图谱视图、阅读更新后的页面。**Obsidian 是 IDE；LLM 是程序员；Wiki 是代码库。**

这种模式可以应用于很多不同的场景。举几个例子：

- **个人成长**：追踪你的目标、健康、心理、自我提升 — 归档日记条目、文章、播客笔记，随时间构建一幅关于你自己的结构化图景。
- **深度研究**：在数周或数月内深入一个主题 — 阅读论文、文章、报告，渐进式地构建一个包含演进论点的综合 Wiki。
- **读书笔记**：逐章归档，为角色、主题、情节线索及其相互关联建立页面。读完后你就拥有一个丰富的伴读 Wiki。想想像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 这样的粉丝 Wiki — 数千个相互链接的页面涵盖角色、地点、事件、语言，由志愿者社区多年构建。你可以在阅读过程中个人构建类似的东西，让 LLM 完成所有交叉引用和维护。
- **商业/团队**：由 LLM 维护的内部 Wiki，以 Slack 对话、会议记录、项目文档、客户通话为输入。可以设置人工审核环节。Wiki 保持最新，因为 LLM 承担了团队中没人愿意做的维护工作。
- **竞品分析、尽职调查、旅行规划、课程笔记、兴趣深挖** — 任何你在持续积累知识并希望它有序组织而非散落各处的场景。

## 架构

分为三层：

### 原始资料层（Raw Sources）

你精心策划的原始文档集合。文章、论文、图片、数据文件。这些是**不可变的** — LLM 从中读取但绝不修改。这是你的**信息源头（Source of Truth）**。

### Wiki 层

一个由 LLM 生成的 Markdown 文件目录。包括摘要、实体页面、概念页面、对比分析、概览、综合分析。**LLM 完全拥有这一层。** 它创建页面、在新来源到达时更新页面、维护交叉引用、并保持一切一致性。你只负责阅读；LLM 负责写。

### 模式层（Schema）

一份文档（如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md），告诉 LLM Wiki 的结构是什么、约定是什么、在摄入来源/回答问题/维护 Wiki 时应遵循什么工作流。这是**关键配置文件** — 它让 LLM 成为一个有纪律的 Wiki 维护者，而不是一个通用聊天机器人。你和 LLM 会随着时间共同演进这个文档，摸索出适合你领域的最佳实践。

## 操作流程

### 摄入（Ingest）

你把一个新来源放入原始资料集合，然后告诉 LLM 处理它。一个典型流程：LLM 阅读来源，与你讨论关键要点，在 Wiki 中写一个摘要页面，更新索引，更新 Wiki 中相关的实体和概念页面，并在日志中追加一条记录。**一个来源可能会触及 10-15 个 Wiki 页面。** 就我个人而言，我倾向于逐个摄入来源并保持参与 — 我阅读摘要、检查更新、指导 LLM 该强调什么。但你也可以一次批量摄入多个来源，减少监督。关键是开发出适合你风格的工作流，并将其记录在模式文档中供未来会话使用。

### 查询（Query）

你针对 Wiki 提问。LLM 搜索相关页面、阅读它们、综合出带引用的答案。答案可以根据问题采取不同形式 — Markdown 页面、对比表格、幻灯片（Marp）、图表（matplotlib）、画布。重要洞察：**好的答案可以作为新页面归档回 Wiki。** 你要求的对比、分析、发现的关联 — 这些都很有价值，不应该消失在聊天历史中。这样你的探索就像摄入的资料一样在知识库中持续复合增值。

### 审查（Lint）

定期让 LLM 对 Wiki 进行健康检查。寻找：页面之间的矛盾、被更新来源已取代的过时论断、没有入链的孤岛页面、被提及但缺少独立页面的重要概念、缺失的交叉引用、可以通过网络搜索填补的数据空白。LLM 善于建议新的研究问题和新的资料来源。这让 Wiki 在增长过程中保持健康。

## 索引与日志

两个特殊文件帮助 LLM（和你）在 Wiki 增长时导航。它们各有不同用途：

### 索引（index.md）

面向**内容**。它是 Wiki 中所有内容的目录 — 每个页面列出链接、一行摘要、以及可选的元数据（如日期或来源数量）。按类别组织（实体、概念、来源等）。LLM 在每次摄入时更新它。回答查询时，LLM 先读索引找到相关页面，再深入阅读。这在中等规模（约 100 个来源、数百个页面）下效果出奇地好，避免了对基于嵌入向量的 RAG 基础设施的需求。

### 日志（log.md）

面向**时间线**。它是一个只追加的记录，记载发生了什么以及何时发生 — 摄入、查询、审查。一个有用的技巧：如果每个条目都以一致的前缀开头（如 `## [2026-04-02] ingest | 文章标题`），日志就可以用简单的 Unix 工具解析 — `grep "^## \[" log.md | tail -5` 就能给你最近 5 条记录。日志为你提供 Wiki 演进的时间线，帮助 LLM 了解最近做了什么。

## 可选：CLI 工具

到一定阶段你可能想构建一些小工具，帮助 LLM 更高效地操作 Wiki。Wiki 页面的搜索引擎是最明显的需求 — 小规模时索引文件就够了，但 Wiki 增长后你需要正式的搜索。[qmd](https://github.com/tobi/qmd) 是个好选择：它是一个本地 Markdown 文件搜索引擎，支持混合 BM25/向量搜索和 LLM 重排序，完全在设备上运行。它既有 CLI（LLM 可以调用 shell）又有 MCP 服务器（LLM 可以作为原生工具使用）。你也可以自己构建更简单的东西 — LLM 可以帮你快速编写一个朴素的搜索脚本。

## 技巧与窍门

- **Obsidian Web Clipper** 是一个浏览器扩展，可以把网页文章转换为 Markdown。非常适合快速将资料来源导入你的原始资料集合。
- **本地下载图片。** 在 Obsidian 设置 → 文件与链接 中，将"附件文件夹路径"设为固定目录（如 `raw/assets/`）。然后在 设置 → 快捷键 中搜索 "Download"，找到"为当前文件下载附件"并绑定快捷键（如 Ctrl+Shift+D）。剪藏文章后按快捷键，所有图片就会下载到本地。这是可选的但很有用 — 让 LLM 能直接查看和引用图片，而不是依赖可能失效的 URL。注意 LLM 目前无法在一次传入中原生阅读带内嵌图片的 Markdown — 变通方法是让 LLM 先读文本，再单独查看部分或全部引用的图片以获取额外上下文。有点笨拙但效果不错。
- **Obsidian 的图谱视图**是查看 Wiki 形态的最佳方式 — 什么与什么相连、哪些页面是枢纽、哪些是孤岛。
- **Marp** 是一种基于 Markdown 的幻灯片格式。Obsidian 有对应插件。适合直接从 Wiki 内容生成演示文稿。
- **Dataview** 是一个 Obsidian 插件，可对页面 frontmatter 运行查询。如果你的 LLM 给 Wiki 页面添加了 YAML frontmatter（标签、日期、来源数量），Dataview 就能生成动态表格和列表。
- Wiki 就是一个 Markdown 文件的 git 仓库。你免费获得版本历史、分支和协作能力。

## 为什么这个方法有效

维护知识库中最令人厌烦的部分不是阅读或思考 — 而是**簿记工作**。更新交叉引用、保持摘要最新、标注新数据与旧论断的矛盾、在数十个页面间维护一致性。人类之所以放弃 Wiki，是因为维护负担的增长速度超过了价值的增长速度。**LLM 不会感到无聊，不会忘记更新交叉引用，一次操作就能触及 15 个文件。** Wiki 得以持续维护，因为维护成本几乎为零。

**人类的工作**是策划来源、引导分析、提出好问题、思考这一切意味着什么。**LLM 的工作**是其他所有事情。

这个理念在精神上与 Vannevar Bush 的 Memex（1945 年）相通 — 一个个人的、精心策划的知识存储，文档之间有联想式的轨迹。Bush 的愿景比万维网后来的实际形态更接近这里描述的模式：私人的、主动策划的、文档之间的连接和文档本身一样有价值。他无法解决的部分是——谁来做维护。LLM 解决了这个问题。

## 备注

本文档是刻意抽象的。它描述的是理念，而非具体实现。确切的目录结构、模式约定、页面格式、工具链 — 这些都取决于你的领域、偏好和所选的 LLM。上面提到的一切都是可选和模块化的 — 选择有用的，忽略不需要的。例如：你的资料可能全是纯文本，不需要图片处理。你的 Wiki 可能足够小，索引文件就是你所需的全部，不需要搜索引擎。你可能不在乎幻灯片，只想要 Markdown 页面。你可能想要一套完全不同的输出格式。正确的使用方式是把这份文档分享给你的 LLM Agent，然后一起协作，实例化一个适合你需求的版本。这份文档唯一的工作就是传达这个模式。你的 LLM 可以搞定剩下的。
