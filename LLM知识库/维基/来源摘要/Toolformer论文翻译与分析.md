# Toolformer：语言模型可以自学使用工具

> **原文标题**: Toolformer: Language Models Can Teach Themselves to Use Tools
> **作者**: Timo Schick, Jane Dwivedi-Yu, Roberto Dessì, Roberta Raileanu, Maria Lomeli, Luke Zettlemoyer, Nicola Cancedda, Thomas Scialom
> **机构**: Meta AI Research, Universitat Pompeu Fabra
> **发表时间**: 2023年2月9日 (arXiv:2302.04761v1)

---

## 摘要

语言模型（LMs）展现出仅凭少量示例或文本指令就能解决新任务的卓越能力，尤其是在规模较大时。然而，矛盾的是，它们在基本功能上却表现挣扎，例如算术运算或事实查询——而这些恰恰是更简单、更小的模型擅长的领域。

本文展示了语言模型可以通过简单的 API **自学使用外部工具**，从而实现两全其美。我们引入了 **Toolformer**，这是一个经过训练的模型，能够决定：

- 调用哪些 API
- 何时调用它们
- 传递什么参数
- 如何将结果最佳地融入后续的 token 预测

这一切都是以**自监督**的方式完成的，每个 API 只需要少量演示即可。我们整合了多种工具，包括：计算器、问答系统、搜索引擎、翻译系统和日历。

Toolformer 在各种下游任务上实现了显著改进的零样本性能，通常能与更大的模型竞争，同时不牺牲其核心语言建模能力。

## 1. 引言

大型语言模型在各种自然语言处理任务上取得了令人印象深刻的零样本和少样本结果（Brown et al., 2020; Chowdhery et al., 2022 等），并展现出多种涌现能力（Wei et al., 2022）。然而，所有这些模型都有几个**固有的局限性**，这些局限性最多只能通过进一步扩展来部分解决：

| 局限性 | 说明 |
|--------|------|
| 无法获取最新信息 | 无法访问关于近期事件的最新信息（Komeili et al., 2022） |
| 倾向于幻觉 | 容易编造虚假事实（Maynez et al., 2020; Ji et al., 2022） |
| 低资源语言理解困难 | 难以理解资源匮乏的语言（Lin et al., 2021） |
| 缺乏数学技能 | 无法进行精确计算（Patel et al., 2021） |
| 时间感知缺失 | 不知道时间的流逝（Dhingra et al., 2022） |

克服这些局限性的一个简单方法是赋予语言模型使用**外部工具**的能力，如搜索引擎、计算器或日历。然而，现有方法要么依赖大量人工标注（Komeili et al., 2022; Thoppilan et al., 2022），要么将工具使用限制在特定任务设置中（如 Gao et al., 2022; Parisi et al., 2022），阻碍了工具使用在语言模型中的更广泛采用。

### 1.1 Toolformer 的设计目标

因此，我们提出了 Toolformer，一个以新颖方式学习使用工具的模型，满足以下设计要求：

1. **自监督学习工具使用**：无需大量人工标注。这很重要，不仅因为标注成本高昂，还因为**人类认为有用的东西可能与模型认为有用的不同**。

2. **保持通用性**：语言模型不应失去其通用性，应该能够**自行决定何时以及如何使用哪个工具**。与现有方法相比，这使得工具使用更加全面，不局限于特定任务。

### 1.2 核心方法

我们实现这些目标的方法基于最近的思想：使用大型语言模型的上下文学习能力（Brown et al., 2020）从零开始生成整个数据集（Schick and Schütze, 2021b; Honovich et al., 2022; Wang et al., 2022）。

**核心流程**：
1. 给定少量关于如何使用 API 的人工示例
2. 让语言模型用潜在的 API 调用标注大型语言建模数据集
3. 使用自监督损失确定哪些 API 调用实际上帮助模型预测未来 token
4. 在模型认为有用的 API 调用上微调语言模型本身

如图1所示，通过这种简单的方法，语言模型可以学会控制各种工具，并自行选择何时以及如何使用哪个工具。

### 1.3 Toolformer 示例预测

```
【问答系统】
The New England Journal of Medicine is a registered trademark of
[QA("Who is the publisher of The New England Journal of Medicine?")
→ Massachusetts Medical Society] the MMS.

【计算器】
Out of 1400 participants, 400 (or [Calculator(400 / 1400) → 0.29] 29%)
passed the test.

【机器翻译】
The name derives from "la tortuga", the Spanish word for
[MT("tortuga") → turtle] turtle.

【维基百科搜索】
The Brown Act is California's law [WikiSearch("Brown Act") → The Ralph
M. Brown Act is an act of the California State Legislature that
guarantees the public's right to attend and participate in meetings
of local legislative bodies.] that requires legislative bodies, like
city councils, to hold their meetings open to the public.
```

由于我们的方法与所使用的数据集无关，我们可以将其应用于最初用于预训练模型的完全相同的数据集。这确保了模型不会失去其通用性和语言建模能力。

## 2. 方法论

### 2.1 问题定义

我们的目标是通过 API 调用为语言模型 M 配备使用不同工具的能力。我们要求每个 API 的输入和输出都可以表示为文本序列。这允许将 API 调用无缝插入任何给定的文本中，使用特殊 token 标记每个调用的开始和结束。

### 2.2 API 调用表示

我们将每个 API 调用表示为元组 `c = (ac, ic)`，其中：
- `ac` 是 API 的名称
- `ic` 是对应的输入

给定一个 API 调用 c 及其对应结果 r，我们分别将不包含和包含结果的 API 调用线性化序列表示为：

```
e(c) = <API> ac(ic) </API>
e(c, r) = <API> ac(ic) → r </API>
```

其中 `<API>`、`</API>` 和 `→` 是特殊 token。

> **实践注意**：实际上，我们使用 token 序列 `" ["`、`"]"` 和 `"->"` 分别表示 `<API>`、`</API>` 和 `→`。这使我们的方法无需修改现有语言模型的词汇表即可工作。

### 2.3 数据集增强流程

给定一个纯文本数据集 C = {x¹, ..., x^|C|}，我们首先将此数据集转换为一个用 API 调用增强的数据集 C*。这通过三个步骤完成：

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────────────┐
│  LM Dataset │ →  │ 1. 采样     │ →  │ 2. 执行     │ →  │ 3. 过滤             │ →  │ 带API调用的LM数据集 │
│             │    │   API调用   │    │   API调用   │    │   API调用           │    │                     │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────────────┘
```

#### 步骤 1：采样 API 调用

对于每个 API，我们编写一个提示 P(x) 来鼓励语言模型用 API 调用标注示例 x = x₁, ..., xₙ。

设 pM(zₙ₊₁ | z₁, ..., zₙ) 为模型 M 分配给 token zₙ₊₁ 作为序列 z₁, ..., zₙ 的续写的概率。

首先，通过计算以下概率来采样最多 k 个候选 API 调用位置：

```
pᵢ = pM(<API> | P(x), x₁:ᵢ₋₁)
```

即模型 M 在位置 i 开始 API 调用的概率。

给定采样阈值 τs，我们保留所有满足 pᵢ > τs 的位置 I = {i | pᵢ > τs}；如果超过 k 个这样的位置，我们只保留 top k。

对于每个位置 i ∈ I，我们通过从模型 M 采样获得最多 m 个 API 调用 c¹ᵢ, ..., cᵐᵢ，给定序列 [P(x), x₁, ..., xᵢ₋₁, <API>] 作为前缀，</API> 作为序列结束 token。

#### 步骤 2：执行 API 调用

执行模型 M 生成的所有 API 调用以获取相应结果。执行方式完全取决于 API 本身——例如，可以是调用另一个神经网络、执行 Python 脚本或使用检索系统在大型语料库上进行搜索。每个 API 调用 cᵢ 的响应需要是单个文本序列 rᵢ。

#### 步骤 3：过滤 API 调用

设 i 是 API 调用 cᵢ 在序列 x = x₁, ..., xₙ 中的位置，rᵢ 是来自 API 的响应。进一步，给定权重序列 (wᵢ | i ∈ ℕ)，设：

```
Lᵢ(z) = -∑ⱼ₌ᵢⁿ wⱼ₋ᵢ · log pM(xⱼ | z, x₁:ⱼ₋₁)
```

为如果模型以 z 为前缀时，M 在 token xᵢ, ..., xₙ 上的加权交叉熵损失。

我们比较这个损失的两种不同实例化：

```
L⁺ᵢ = Lᵢ(e(cᵢ, rᵢ))
L⁻ᵢ = min(Lᵢ(ε), Lᵢ(e(cᵢ, ε)))
```

其中 ε 表示空序列。

- **L⁺ᵢ**：如果 API 调用及其结果作为前缀提供给 M，则为所有 token xᵢ, ..., xₙ 的加权损失
- **L⁻ᵢ**：从以下两者获得的损失的最小值：(i) 完全不进行 API 调用，(ii) 进行 API 调用但不提供响应

**直觉理解**：如果同时向模型提供 API 调用的输入和输出使模型更容易预测未来 token（相比于完全不接收 API 调用或只接收其输入），则该 API 调用对 M 是有帮助的。

给定过滤阈值 τf，我们只保留满足以下条件的 API 调用：

```
L⁻ᵢ - L⁺ᵢ ≥ τf
```

即添加 API 调用及其结果能够将损失降低至少 τf。

#### 步骤 4：模型微调

在采样和过滤所有 API 的调用后，我们最终合并剩余的 API 调用并将它们与原始输入交错。

对于输入文本 x = x₁, ..., xₙ，在位置 i 有对应的 API 调用和结果 (cᵢ, rᵢ)，我们构建新序列：

```
x* = x₁:ᵢ₋₁, e(cᵢ, rᵢ), xᵢ:ₙ
```

对 C 中所有 x 这样做产生用 API 调用增强的新数据集 C*。我们使用这个新数据集微调 M，使用标准语言建模目标。

**关键点**：除了插入的 API 调用外，增强数据集 C* 包含与 C（原始数据集）完全相同的文本。因此，在 C* 上微调 M 使其接触到与在 C 上微调相同的内容。

此外，由于 API 调用恰好插入在那些帮助 M 预测未来 token 的位置，并使用恰好那些输入，在 C* 上微调使语言模型能够纯粹基于自己的反馈决定何时以及如何使用哪个工具。

### 2.4 推理过程

使用我们的方法微调后生成文本时，我们执行常规解码直到 M 产生 `→` token，表示它接下来期望 API 调用的响应。

此时，我们：
1. 中断解码过程
2. 调用相应的 API 获取响应
3. 在插入响应和 `</API>` token 后继续解码过程

## 3. 工具介绍

我们探索了各种工具来解决常规语言模型的不同缺陷。我们对这些工具施加的唯一约束是：
1. 它们的输入和输出都可以表示为文本序列
2. 我们可以获得少量关于其预期使用的演示

### 3.1 工具概览

| API 名称 | 示例输入 | 示例输出 |
|----------|----------|----------|
| **问答系统** | Where was the Knights of Columbus founded? | New Haven, Connecticut |
| **维基百科搜索** | Fishing Reel Types | Spin fishing > Spin fishing is distinguished between fly fishing and bait cast fishing by the type of rod and reel used... |
| **计算器** | 27 + 4 * 2 | 35 |
| **日历** | ε（空） | Today is Monday, January 30, 2023. |
| **机器翻译** | sûreté nucléaire | nuclear safety |

### 3.2 各工具详情

#### 问答系统（Question Answering）

我们的第一个工具是基于另一个语言模型的问答系统，可以回答简单的事实性问题。

具体来说，我们使用 **Atlas**（Izacard et al., 2022），这是一个在 Natural Questions（Kwiatkowski et al., 2019）上微调的检索增强语言模型。

#### 计算器（Calculator）

作为第二个工具，我们使用一个可以执行简单数值计算的计算器。

- 只支持四种基本算术运算：+、-、*、/
- 结果始终保留两位小数

#### 维基百科搜索（Wikipedia Search）

第三个工具是一个搜索引擎，给定搜索词，返回来自维基百科的简短文本片段。

与我们的问答工具相比，此搜索使模型能够获取关于某个主题的更全面信息，但需要模型自己提取相关部分。

作为搜索引擎，我们使用 **BM25 检索器**（Robertson et al., 1995; Baeza-Yates et al., 1999）来索引来自 KILT（Petroni et al., 2021）的维基百科转储。

#### 机器翻译系统（Machine Translation System）

第四个工具是基于语言模型的机器翻译系统，可以将任何语言的短语翻译成英语。

- 使用 **600M 参数的 NLLB**（Costa-jussà et al., 2022）作为多语言机器翻译模型
- 支持 200 种语言（包括低资源语言）
- 使用 **fastText 分类器**（Joulin et al., 2016）自动检测源语言
- 目标语言始终设置为英语

#### 日历（Calendar）

最后一个工具是日历 API，查询时返回当前日期，不接收任何输入。这为需要某些时间感知的预测提供时间上下文。

## 4. 实验

我们研究我们的方法是否使模型能够在没有任何进一步监督的情况下使用工具，并自行决定何时以及如何调用哪个可用工具。

### 4.1 实验设置

#### 数据集生成

在所有实验中，我们使用：
- **CCNet**（Wenzek et al., 2020）的子集作为语言建模数据集 C
- **GPT-J**（Wang and Komatsuzaki, 2021）作为语言模型 M

为了降低用 API 调用标注 C 的计算成本，我们为某些 API 定义启发式规则，以获取 C 的子集，对这些子集来说 API 调用可能比普通文本更有帮助。例如，我们只考虑包含至少三个数字的文本用于计算器工具。

#### 数据集统计

| API | τf = 0.5 | τf = 1.0 | τf = 2.0 |
|-----|----------|----------|----------|
| 问答系统 | 51,987 | 18,526 | 5,135 |
| 维基百科搜索 | 207,241 | 60,974 | 13,944 |
| 计算器 | 3,680 | 994 | 138 |
| 日历 | 61,811 | 20,587 | 3,007 |
| 机器翻译 | 3,156 | 1,034 | 229 |

*表：不同过滤阈值 τf 下 C* 中带有 API 调用的示例数量*

#### 权重函数

对于权重函数，我们使用：

```
wₜ = w̃ₜ / ∑ₛ∈ℕ w̃ₛ，其中 w̃ₜ = max(0, 1 - 0.2·t)
```

以确保 API 调用发生在 API 提供的信息对模型实际有帮助的位置附近。

#### 模型微调

我们在 C* 上微调 M，使用：
- 批大小：128
- 学习率：1·10⁻⁵
- 前 10% 训练步骤线性预热

#### 基线模型

| 模型 | 描述 |
|------|------|
| **GPT-J** | 未经任何微调的常规 GPT-J 模型 |
| **GPT-J + CC** | 在 C（不含任何 API 调用的 CCNet 子集）上微调的 GPT-J |
| **Toolformer** | 在 C*（用 API 调用增强的 CCNet 子集）上微调的 GPT-J |
| **Toolformer (disabled)** | 与 Toolformer 相同，但解码时禁用 API 调用 |
| **OPT (66B)** | 约 10 倍于基线模型的大型模型 |
| **GPT-3 (175B)** | 约 25 倍于基线模型的大型模型 |

### 4.2 下游任务

我们在各种下游任务上评估所有模型。在所有情况下，我们考虑**提示零样本设置**——即模型被指示用自然语言解决每个任务，但我们不提供任何上下文示例。

这与之前的工具使用工作形成对比（如 Gao et al., 2022; Parisi et al., 2022），在那些工作中模型被提供了关于如何使用工具解决具体任务的特定数据集示例。

我们选择更具挑战性的零样本设置，因为我们感兴趣的是看 Toolformer 是否在用户没有预先指定应该以哪种方式使用哪些工具来解决特定问题的情况下工作。

**解码策略修改**：
- 不仅当 `<API>` 是最可能的 token 时生成它，而是当它是 k 个最可能的 token 之一时
- 使用 k = 10 以增加模型使用其可访问 API 的倾向
- 每个输入最多只允许一次 API 调用，以确保模型不会陷入不断调用 API 而不产生任何实际输出的循环

#### 4.2.1 LAMA 基准测试

我们在 LAMA 基准测试（Petroni et al., 2019）的 SQuAD、Google-RE 和 T-REx 子集上评估模型。对于每个子集，任务是用缺失的事实（如日期或地点）完成一个简短陈述。

| 模型 | SQuAD | Google-RE | T-REx |
|------|-------|-----------|-------|
| GPT-J | 17.8 | 4.9 | 31.9 |
| GPT-J + CC | 19.2 | 5.6 | 33.2 |
| Toolformer (disabled) | 22.1 | 6.3 | 34.9 |
| **Toolformer** | **33.8** | **11.5** | **53.5** |
| OPT (66B) | 21.6 | 2.9 | 30.1 |
| GPT-3 (175B) | 26.8 | 7.0 | 39.8 |

**关键发现**：
- Toolformer 明显优于所有相同大小的基线模型，分别提高了 11.7、5.2 和 18.6 个点
- 明显优于 OPT (66B) 和 GPT-3 (175B)，尽管这两个模型大得多
- 模型在几乎所有情况下（98.1%）独立决定向问答工具询问所需信息
- 只有很少的示例使用不同的工具（0.7%）或根本不使用工具（1.2%）

#### 4.2.2 数学数据集

我们在 ASDiv（Miao et al., 2020）、SVAMP（Patel et al., 2021）和 MAWPS 基准测试（Koncel-Kedziorski et al., 2016）上测试数学推理能力。

| 模型 | ASDiv | SVAMP | MAWPS |
|------|-------|-------|-------|
| GPT-J | 7.5 | 5.2 | 9.9 |
| GPT-J + CC | 9.6 | 5.0 | 9.3 |
| Toolformer (disabled) | 14.8 | 6.3 | 15.0 |
| **Toolformer** | **40.4** | **29.4** | **44.0** |
| OPT (66B) | 6.0 | 4.9 | 7.9 |
| GPT-3 (175B) | 14.0 | 10.0 | 19.8 |

**关键发现**：
- GPT-J 和 GPT-J + CC 表现大致相同
- 即使禁用 API 调用，Toolformer 也能取得更好的结果（推测是因为模型在许多 API 调用及其结果的示例上进行了微调，提高了其自身的数学能力）
- 允许模型进行 API 调用使所有任务的性能提高了一倍以上
- 明显优于更大的 OPT 和 GPT-3 模型
- 在所有基准测试中，97.9% 的示例模型决定向计算器工具求助

#### 4.2.3 问答任务

我们查看 Web Questions（Berant et al., 2013）、Natural Questions（Kwiatkowski et al., 2019）和 TriviaQA（Joshi et al., 2017）。

| 模型 | WebQS | NQ | TriviaQA |
|------|-------|-----|----------|
| GPT-J | 18.5 | 12.8 | 43.9 |
| GPT-J + CC | 18.4 | 12.2 | 45.6 |
| Toolformer (disabled) | 18.9 | 12.6 | 46.7 |
| **Toolformer** | **26.3** | **17.7** | **48.8** |
| OPT (66B) | 18.6 | 11.4 | 45.7 |
| GPT-3 (175B) | 29.0 | 22.6 | 65.9 |

**关键发现**：
- Toolformer 明显优于所有其他基于 GPT-J 的模型
- 主要依赖维基百科搜索 API（99.3%）来查找相关信息
- 仍落后于更大的 GPT-3 (175B) 模型
- 可能原因：搜索引擎简单（许多情况下返回与给定查询明显不匹配的结果）以及 Toolformer 无法与其交互（如重新表述查询或浏览多个顶部结果）

#### 4.2.4 多语言问答

我们在 MLQA（Lewis et al., 2019）上评估 Toolformer 和所有基线模型，这是一个多语言问答基准测试。

### 4.3 语言建模

除了验证各种下游任务的改进性能外，我们还想确保 Toolformer 的语言建模性能不会因我们使用 API 调用的微调而下降。

|------|----------|-------|

| Toolformer (disabled) | 10.3 | **10.5** |

**关键发现**：
- 在 CCNet 上微调导致在不同 CCNet 子集上性能略有提高
- 但在 WikiText 上性能略有下降（可能因为 GPT-J 的原始预训练数据更类似于 WikiText）
- **最重要的是**：在 C*（用 API 调用标注的数据集）上训练不会导致困惑度增加

### 4.4 缩放定律

我们研究请求外部工具帮助的能力如何随语言模型大小变化而受影响。为此，我们不仅将方法应用于 GPT-J，还应用于四个较小的 GPT-2 家族模型（124M、355M、775M 和 1.6B 参数）。

**关键发现**：
- 利用提供的工具的能力**只在约 775M 参数时才开始涌现**
- 较小的模型无论是否使用工具都表现相似
- 例外是维基百科搜索引擎（主要用于 QA 基准测试）——假设这是因为该 API 相对容易使用
- 随着模型规模增大，它们在没有 API 调用的情况下解决任务的能力提高
- 但同时它们善用提供的 API 的能力也在提高
- 结果是即使对最大的模型，有无 API 调用的预测之间仍有很大差距

## 6. 相关工作

### 6.1 语言模型预训练

有各种方法在预训练期间用某种形式的额外文本信息增强语言模型，包括：
- 各种形式的元数据（Keskar et al., 2019）
- HTML 标签（Aghajanyan et al., 2021）
- 维基百科标记（Schick et al., 2022）
- 从信息检索系统获得的相关文本（Guu et al., 2020; Borgeaud et al., 2021; Izacard et al., 2022）

对于所有这些方法，无论是否有帮助，都会**始终提供**额外信息。相比之下，Toolformer 学会**明确请求**正确的信息。

### 6.2 工具使用

几种方法旨在为语言模型配备使用外部工具的能力，如：
- 搜索引擎（Komeili et al., 2022; Thoppilan et al., 2022; Lazaridou et al., 2022; Shuster et al., 2022; Yao et al., 2022）
- 网络浏览器（Nakano et al., 2021）
- 计算器（Cobbe et al., 2021; Thoppilan et al., 2022）
- 翻译系统（Thoppilan et al., 2022）
- Python 解释器（Gao et al., 2022）

这些模型学习使用工具的方式大致可以分为两种方法：
1. **依赖大量人工监督**（Komeili et al., 2022; Nakano et al., 2021; Thoppilan et al., 2022）
2. **通过针对特定任务的少样本设置提示语言模型**，事先知道需要使用哪个工具（Gao et al., 2022; Lazaridou et al., 2022; Yao et al., 2022）

相比之下，Toolformer 的**自监督性质**使其能够学习如何以及何时使用工具，而不需要显示任务特定示例的特定提示。

与我们工作最相关的可能是 **TALM**（Parisi et al., 2022），这是一种使用类似自监督目标教模型使用计算器和搜索引擎的方法，但只在模型针对下游任务微调的设置中探索。

### 6.3 自举技术

使用自训练和自举技术改进模型的想法已在各种上下文中研究，包括：
- 词义消歧（Yarowsky, 1995）
- 关系抽取（Brin, 1999; Agichtein and Gravano, 2000）
- 解析（McClosky et al., 2006; Reichart and Rappoport, 2007）
- 序列生成（He et al., 2020）
- 少样本文本分类（Schick and Schütze, 2021a）
- 检索（Izacard and Grave, 2021）
- 推理（Zelikman et al., 2022）

与这些方法类似，Toolformer 在应用基于困惑度的过滤步骤后，在自己的预测上进行训练。

## 7. 局限性

虽然我们的方法使语言模型能够以自监督方式学习如何使用各种工具，但当前形式的方法存在一些明显的局限性：

### 7.1 无法链式使用工具

Toolformer 目前**无法链式使用工具**（即使用一个工具的输出作为另一个工具的输入）。这是因为每个工具的 API 调用是独立生成的；因此，微调数据集中没有链式工具使用的示例。

### 7.2 无法交互式使用工具

我们当前的方法也**不允许语言模型以交互方式使用工具**——特别是对于像搜索引擎这样可能返回数百个不同结果的工具，使语言模型能够浏览这些结果或以类似于 Nakano et al. (2021) 的方式优化其搜索查询对于某些应用可能至关重要。

### 7.3 对输入措辞敏感

我们发现使用 Toolformer 训练的模型在决定是否调用 API 时，通常**对其输入的确切措辞非常敏感**；鉴于语言模型已知在零样本和少样本设置中对提供的提示非常敏感（Jiang et al., 2020; Schick and Schütze, 2021a），这或许不令人惊讶。

### 7.4 样本效率低

根据工具不同，我们的方法也可能**非常低效**；例如，处理超过一百万个文档只产生几千个有用的计算器 API 调用示例。

**潜在解决方案**：可能是迭代应用我们的方法，类似于相关自举方法中的做法（Schick and Schütze, 2021a; Izacard and Grave, 2021; Parisi et al., 2022）。

### 7.5 不考虑计算成本

最后，在决定是否进行 API 调用时，Toolformer 目前**不考虑进行 API 调用所产生的工具相关计算成本**。

---

## 8. 结论

我们引入了 **Toolformer**，这是一个以自监督方式学习如何使用不同工具（如搜索引擎、计算器和翻译系统）通过简单 API 调用的语言模型。

这是通过在大量采样的 API 调用上微调实现的，这些调用根据是否降低未来 token 的困惑度进行过滤。

**核心成就**：
- Toolformer 显著提高了 6.7B 参数 GPT-J 模型的零样本性能
- 使其甚至能够在一系列不同的下游任务上超越更大的 GPT-3 模型
